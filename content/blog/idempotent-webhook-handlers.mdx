Webhook delivery is *at-least-once*: providers retry, networks hiccup, and the same event lands twice more often than you'd think. Without a guard, that's a duplicate invoice or a double-sent email. One line of Redis stops it.

## The guard

```ts
// idempotency.ts
const key = `evt:${event.id}`;
if (await redis.set(key, 1, "NX", "EX", 86400) === null)
  return io.logger.info("duplicate, skipped");
```

`SET … NX` only succeeds the first time it sees an event id. Every retry after that returns `null`, so the handler bails before doing any work.

## Getting it right

- Set the key **before** the side effect, not after.
- Make the TTL at least as long as the provider's retry window, 24h is a safe default.
- Use the provider's event id when there is one; otherwise hash the stable fields of the payload, never the timestamp.

It's the cheapest reliability win in the whole stack, and every public webhook deserves it.
