Catching Trigger.dev webhook timeouts gracefully

Trigger.dev Webhooks Node.js
C Cristi Șatcovschi
I build automation infrastructure for small teams.

Every webhook handler eventually calls something slow — a CRM, a payments API, a model endpoint. When that call hangs, Trigger.dev doesn't wait politely: the whole run sits open, burns execution time, and eventually fails the delivery. The fix isn't a bigger timeout. It's bounding the call so a slow dependency degrades gracefully instead of taking the run down with it.

The failure mode

A naive handler awaits the API call directly. If HubSpot takes thirty seconds to respond, your run takes thirty seconds too — and if it never responds, the run hangs until the platform kills it. You get a failed delivery, no retry, and a payload you now have to replay by hand.

Wrap the call in a bounded task

runTask is the unit of reliability in Trigger.dev. Give it a timeoutInMs and the call is cancelled the moment it overruns — the task fails cleanly, the run doesn't.

src/trigger/hubspot-sync.ts 
client.defineJob({
  id: "hubspot-sync",
  trigger: eventTrigger({ name: "contact.created" }),
  run: async (payload, io) => {
    // bound the slow call so it can't hang the run
    const contact = await io.runTask(
      "fetch-contact",
      { timeoutInMs: 8000 },
      () => hubspot.contacts.get(payload.id),
    );
    await io.sendEvent("synced", { payload: contact });
  },
});

Because the fetch lives inside its own task, Trigger.dev checkpoints the work around it. If the run is resumed, the steps that already completed aren't repeated.

Let it retry itself

A timeout usually means the upstream is briefly overloaded, not broken. Add a retry policy and the task backs off and tries again before it gives up — no plumbing of your own.

retry-policy.ts 
await io.runTask(
  "fetch-contact",
  {
    timeoutInMs: 8000,
    retry: { limit: 4, factor: 2, minTimeoutInMs: 1000 },
  },
  () => hubspot.contacts.get(payload.id),
);

Four attempts with exponential backoff cover almost every transient blip — the kind that would otherwise page you at 2am.

Why this matters

  • A slow dependency degrades one task, not the entire workflow.
  • Transient failures self-heal — no manual payload replays.
  • Execution time stays bounded, so your bill stays predictable.

That's the whole pattern: bound the call, let it retry, move on. Ship it once and every webhook you wire up afterwards inherits the same safety net.
